# LayoutSans v0.2 — Interactive Text Features

This document describes the features and capabilities being added in version 0.2 of LayoutSans.

## Overview

Version 0.2 transforms LayoutSans from a static layout engine into a fully interactive text system while maintaining the core principle: zero visual DOM layout. All interactivity is rendered on canvas with minimal semantic DOM for accessibility and OS integration.

## Core Features

### Text Selection

Users can select text using mouse or touch input, just like in a browser:

- **Desktop**: Click and drag to select text across multiple text nodes and line wraps
- **Mobile**: Long-press to select a word, then drag native OS handles to adjust selection
- **Keyboard**: Shift+Arrow keys to extend selection, Ctrl/Cmd+A to select all
- **Visual feedback**: Selection highlights rendered on canvas with platform-appropriate colors

### Clipboard Integration

Selected text can be copied to the system clipboard:

- **Desktop**: Ctrl+C / Cmd+C copies selected text
- **Mobile**: Native "Copy" option appears in OS context menu
- **Programmatic**: API to copy text programmatically
- **Cross-platform**: Works consistently across all browsers and operating systems

### Interactive Hyperlinks

Links are clickable and keyboard-navigable:

- **Click navigation**: Click links to navigate (respects target and rel attributes)
- **Keyboard navigation**: Tab through links in document order, Enter/Space to activate
- **Visual feedback**: Hover cursor changes to pointer, focus rings for keyboard navigation
- **Accessibility**: Links are properly exposed to screen readers

### Full-Text Search

Built-in search functionality similar to browser Ctrl+F:

- **Keyboard shortcut**: Ctrl+F / Cmd+F opens search panel
- **Match highlighting**: All matches highlighted on canvas in yellow, active match in orange
- **Navigation**: Ctrl+G / Cmd+G to jump between matches
- **Smooth scrolling**: Animated scroll to bring matches into view
- **Match counter**: Shows "3 of 47" style match count

### Screen Reader Support

Full accessibility for users with visual impairments:

- **Semantic structure**: Headings, paragraphs, and links properly exposed
- **Document order**: Content announced in correct reading order
- **Keyboard navigation**: All interactive elements accessible via keyboard
- **Live regions**: Dynamic content changes announced appropriately
- **Skip navigation**: Skip link for quick content access

### Spatial Hit-Testing

Efficient point queries for mouse/touch interaction:

- **Fast lookups**: O(log n) query time using R-Tree spatial index
- **Sub-character precision**: Resolves clicks to exact character positions
- **Scalable**: Maintains performance even with 100,000+ items

## New Node Types

### LinkNode

Represents clickable hyperlinks:

```typescript
{
  type: 'link'
  href: string
  target?: '_blank' | '_self' | '_parent' | '_top'
  rel?: string
  aria?: { label?: string }
  children?: Node[]
}
```

### HeadingNode

Represents semantic headings (h1-h6):

```typescript
{
  type: 'heading'
  level: 1 | 2 | 3 | 4 | 5 | 6
  content: string
  font?: string
  lineHeight?: number
}
```

## API Additions

### Engine Methods

- `buildIndex()` - Build spatial index for hit-testing (async, runs in idle time)
- `mount(container, canvas)` - Mount interaction layer and accessibility tree
- `scrollTo(nodeId, behavior?)` - Scroll to a specific node
- `getAllRecords()` - Get all box records in tree order
- `getTextLineData(nodeId)` - Get text line data for a text node
- `extractText()` - Get plain text of entire layout
- `setSelection(...)` - Programmatically set text selection
- `clearSelection()` - Clear current selection
- `copySelectedText()` - Copy selection to clipboard

### Properties

- `engine.selection` - Current selection state
- `engine.layoutSearch` - Search controller

### Options

- `searchUI` - Enable/disable built-in search panel
- `selectionColor` - Custom selection highlight color
- `searchHighlightColor` - Custom search match color
- `onLinkClick` - Custom link click handler
- `onSelectionChange` - Selection change callback

## Architecture Principles

### Zero Visual DOM Layout

The DOM is used only for:
- **Accessibility**: Semantic tree for screen readers (O(viewport) nodes)
- **OS integration**: Single proxy element for clipboard and mobile handles
- **Search UI**: Optional search panel (can be disabled)

All visual rendering happens on canvas. No DOM elements are created for layout purposes.

### Performance Guarantees

- Layout computation time: unchanged from v0.1 (< 5ms for 100k items)
- Spatial index build: < 15ms (runs in idle time, non-blocking)
- Hit-test query: < 0.5ms
- Selection repaint: < 1ms
- Frame time budget: < 3ms total

### Scalability

DOM node count remains constant regardless of total item count:
- 100 items: ~50 semantic nodes
- 100,000 items: ~600 semantic nodes (viewport-dependent only)

## Browser Support

- Firefox 63+

No polyfills required. Graceful fallbacks for older browsers.

## Migration from v0.1

Version 0.2 is fully backward compatible. Existing v0.1 code continues to work without changes.

To enable new features:

1. Call `buildIndex()` after `compute()`
2. Call `mount()` to enable interaction
3. Call `bridge.sync()` in your render loop

All new features are opt-in. Static rendering continues to work exactly as before.

## What's Not Included

The following are planned for future versions:

- RTL (right-to-left) text layout → v0.4
- Advanced grid features (template areas, named lines) → v0.3
- Aspect ratio constraints → v0.3
- Baseline alignment → v0.4

## Summary

v0.2 makes LayoutSans a complete interactive text system while preserving its core advantages: no DOM layout overhead, pure TypeScript, works everywhere JavaScript runs, and maintains the performance characteristics that make it suitable for rendering 100,000+ items.

Users get native-feeling text selection, clipboard integration, clickable links, search, and full accessibility — all rendered on canvas with minimal DOM footprint.
